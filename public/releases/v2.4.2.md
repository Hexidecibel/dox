## Phase 2 — Capture Reviewer Decisions

Every action a reviewer takes in the queue now writes a structured
record so the learning loop has training data to work with.

### Added
- **`reviewer_field_picks`, `reviewer_field_dismissals`,
  `reviewer_table_edits`** tables (migration 0038). Tenant-scoped, FK
  to `processing_queue` with `ON DELETE CASCADE`.
- **`extraction_examples` dedupe.** Added `input_text_hash` plus a
  partial unique index so future writes don't accumulate duplicates.
- **Per-row chevron buttons** on the compare panel — reviewers can
  take individual fields from one side without globally adopting either
  text or VLM.
- **25 new tests** — 19 unit on the pure capture builder, 6 API on
  the approve-time capture wiring.

### Changed
- **Approve payload** (single + multi product) now accepts
  `field_picks`, `dismissals`, `table_edits` arrays.
  `persistReviewerCaptures` inserts each with per-row try/catch so a
  capture failure can't block the approve.
- **ReviewQueue** derives per-field picks at approve time by diffing
  the edited fields against text and VLM payloads. Dismissals and
  table edits — previously visual-only — now flow to the backend.

### Fixed
- **VLM-only items** correctly initialize `vlmSelectedSource` and
  `editedFields` from the VLM payload, fixing the prior bug where
  approve fired with an empty `primary_metadata`.

### Internal
- Phase 2 of the breezy-hatching-moonbeam plan — the data plane for
  pre-fill and confidence-aware UX in Phase 3a.
